## 3. 위협 시나리오 구체화 (Concrete Threat Scenarios)

> 원본 3항은 "통제절차·보안검증 없이 업무PC에서 작동되는 프로그램을 만들 수 있다"는 추상 수준. 경영진 설득을 위해 구체 시나리오 3~4개 필요.

### T1. 비인가 데이터 추출 스크립트
- **구도**: 직원이 Geni-Code에게 "엑셀 고객 리스트에서 이메일만 뽑아 CSV로 저장" 요청 → 파이썬 스크립트 생성 → 업무PC에서 직접 실행
- **리스크**: 변경관리/DLP를 거치지 않은 경로로 **개인정보 파일 생성** 및 이후 외부 반출 시도
- **영향**: 개인정보보호법 위반, 금융실명법 위반 가능성

### T2. 쉐도우 자동화 (Shadow IT / Shadow Automation)
- **구도**: 업무 반복 절차를 Geni-Code로 자동화 스크립트 작성 → 스케줄러(Task Scheduler/cron)로 상시 구동
- **리스크**: 통제 외 자동화가 **정식 시스템과 병존**. 오작동 시 원장·계정계 연쇄 영향 가능
- **영향**: 장애 원인 추적 불가, 감사 시 감점, 책임 소재 불명확

### T3. 악성 라이브러리 유입 (공급망)
- **구도**: Geni-Code가 제안한 파이썬 패키지(PyPI)에 타이포스쿼팅 또는 트로이화된 버전 존재 → 내부 오프라인 미러 경유 유입
- **리스크**: 의존성 해시 검증 없이 설치되면 **업무PC에 악성 코드 상주**
- **영향**: 계정 탈취, 내부 네트워크 정찰 발판
- **참고**: 실제 PyPI 공격 사례(`colorama` 변종, `ctx`/`PyKafka` 등) 다수 보고됨

### T4. 실행 환경 오염 (권한 상승)
- **구도**: Geni-Code 생성 스크립트에 `os.system`, `subprocess`, PowerShell 호출이 포함 → 직원이 무심코 실행
- **리스크**: 기존 앱 화이트리스트가 Python 인터프리터 한 번만 허용해도 **Python을 로더로 삼아 임의 코드 실행**
- **영향**: 앱 화이트리스트 정책의 실질 무력화 (LoLBAS/LoLBin 유사 효과)

## 4. 핵심 논리 구조 재정리

원본을 논리적으로 재배열하면 다음과 같다:

```
[전제 1] IBK 시스템은 개발-QA-운영 + 결재 + 변경관리로 통제된다
[전제 2] 이 통제는 작성 주체(사람/Geni-Code/AIDD)와 무관하게 적용된다
  ∴ 통제 경로를 통과한 Geni-Code 산출물은 보안상 안전하다.

[전제 3] Geni-Code 사용을 위해 업무PC에 Python 실행을 허용해야 한다
[전제 4] 업무PC는 본래 "허용된 프로그램만 실행" 원칙으로 통제된다
  ∴ Python 허용은 앱 화이트리스트 원칙의 예외이며,
     직원이 결재·QA·변경관리 외부에서 실행 가능한 프로그램을
     직접 작성·실행할 수 있는 경로를 신설한다.

[결론] 따라서 기술적 통제(망·앱 화이트리스트)만으로는 부족하며,
       행위적 통제(보안 서약)와 관리적 통제(실행환경 제한·로그·교육)를
       병행해야 한다.
```
