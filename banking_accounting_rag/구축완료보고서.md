## 1. 프로젝트 개요

은행회계해설(상/하) 2018개정판 PDF 문서를 기반으로, 사용자 질문에 대해 관련 내용을 검색하고 답변하는 **RAG(Retrieval-Augmented Generation) 서비스**를 구축하였다. 로컬 LLM(Ollama)뿐 아니라 **Gemini API, OpenAI API를 UI에서 선택**하여 사용할 수 있는 멀티 LLM 백엔드 구조를 갖추었다. 임베딩은 로컬 bge-m3로 통일하여 벡터 DB를 단일 컬렉션으로 공유한다.

## 3. 구축 과정

### 3.1 사전작업: PDF → 마크다운 변환

- **도구:** PyMuPDF (`fitz`)를 사용한 `scripts/convert_pdf_to_md.py`
- **결과:**
  - 은행회계해설(상): 788페이지 → 1.6MB 마크다운
  - 은행회계해설(하): 564페이지 → 1.2MB 마크다운
- 각 페이지를 `<!-- Page N -->` 주석으로 구분하여 출처 추적 가능

### 3.2 환경 구성

**Ollama v0.19.0 설치 및 모델 다운로드**
- bge-m3 (임베딩, 1.2GB) — 다국어 임베딩 모델, 한국어 지원 우수
- gemma3:12b (LLM, 8.1GB) — 범용 다국어 모델

**Python 패키지**
```
flask, langchain, langchain-core, langchain-text-splitters,
langchain-community, langchain-chroma, chromadb, ollama,
python-dotenv, google-genai, openai
```

### 3.3 문서 전처리 및 청킹

- **1차 분할:** `<!-- Page N -->` 기준 페이지 단위 파싱
- **2차 분할:** `RecursiveCharacterTextSplitter` (1,000자 청크, 200자 오버랩)
- **결과:**
  - 상권: 780페이지 로딩
  - 하권: 547페이지 로딩
  - **총 1,327 페이지 → 1,761개 청크 생성**
- 각 청크에 메타데이터 부착: 권(상/하), 페이지 번호, 청크 인덱스

### 3.4 임베딩 및 벡터 DB 구축

- 임베딩 모델: bge-m3 (Ollama embed API) — 모든 모드에서 공통 사용
- 배치 크기: 20 (서버 안정성을 위해 50에서 축소)
- 재시도 로직: 최대 3회, 지수 백오프
- ChromaDB: cosine 유사도, 단일 컬렉션 `bank_accounting`, 영속 저장 (`./chroma_db/`)
- **1차 구축:** 약 45분 (1,761개 청크) — ChromaDB HNSW 인덱스 오류로 폐기
- **2차 구축 (2026-03-31):** 약 50분 (2,991.5초), 1,761개 청크 정상 완료

### 3.5 RAG 파이프라인

**처리 흐름:**
1. 사용자 질문 → bge-m3로 임베딩 (모든 모드 공통)
2. ChromaDB 단일 컬렉션에서 유사 청크 top-5 검색
3. 검색 결과를 컨텍스트로 프롬프트 구성
4. **현재 선택된 모드의 LLM**으로 답변 생성 (스트리밍)
5. 답변 + 출처(상/하권, 페이지) 반환

**시스템 프롬프트 핵심 규칙:**
- 문서 내용만을 근거로 답변
- 문서에 없는 내용은 명시적으로 안내
- 답변 마지막에 출처 표기

### 3.6 웹 인터페이스

- 채팅 형태 UI (질문 입력 → 실시간 스트리밍 답변)
- SSE(Server-Sent Events) 기반 실시간 토큰 출력
- 답변 하단 출처(권, 페이지) 표시
- 최근 3턴 대화 이력 유지
- **헤더에 LLM 모드 선택 라디오 버튼 (로컬/Gemini/OpenAI)**

**API 엔드포인트:**

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET | `/` | 채팅 페이지 |
| POST | `/ask` | 질문 전송 및 답변 반환 (비스트리밍) |
| POST | `/stream` | SSE 스트리밍 응답 |
| GET | `/get_mode` | 현재 모드 및 사용 가능 모드 조회 |
| POST | `/set_mode` | 모드 변경 (local/gemini/openai) |

### 3.7 멀티 LLM 백엔드 (2026-03-31 추가)

**config.py:**
- `MODE` 설정 (local / gemini / openai) — LLM만 전환
- `EMBED_MODEL = "bge-m3"` — 모든 모드 공통 임베딩
- `COLLECTION_NAME = "bank_accounting"` — 단일 컬렉션
- `.env` 파일로 API 키 관리

**rag_chain.py LLM 모드별 분기:**
- `get_embeddings()` — 항상 Ollama bge-m3 (모드 무관)
- `ask_stream()` — 모드에 따라 Ollama / Gemini / OpenAI LLM 스트리밍
- `set_mode()` / `get_mode()` — 런타임 모드 전환 (서버 재시작 불필요)

**Gemini API:** `google-genai` 패키지, `Client.models.generate_content_stream()` 스트리밍
**OpenAI API:** `openai` 패키지, `ChatCompletion.create(stream=True)` 스트리밍

**UI 모드 선택:**
- 헤더에 로컬 / Gemini / OpenAI 라디오 버튼
- API 키 미설정 시 해당 모드 비활성화 + 사유 표시
- 모드 전환 시 채팅 영역에 안내 메시지

```bash
# 1. 환경 설정

# .env 파일에 API 키 입력 (Gemini/OpenAI 사용 시)

# http://localhost:5000
```

## 6. 구축 중 이슈 및 해결

| 이슈 | 원인 | 해결 방법 |
|------|------|----------|
| `langchain.schema` import 오류 | Python 3.14 + 최신 LangChain 패키지 구조 변경 | `langchain_core.documents`, `langchain_text_splitters`로 import 경로 변경 |
| 임베딩 중 Ollama 서버 연결 끊김 | gemma3 다운로드와 임베딩 동시 실행 시 서버 과부하 | 배치 크기 50→20 축소, 재시도 로직(3회, 지수 백오프) 추가 |
| gemma3:12b 다운로드 타임아웃 | 8.1GB 파일, 네트워크 속도 저하(~400KB/s) | CLI 타임아웃 한계로 별도 터미널에서 직접 다운로드 |
| ChromaDB HNSW 인덱스 오류 | 한글 경로명에서 ChromaDB 인덱스 불안정 | `chroma_db/` 삭제 후 재구축, 프로젝트 폴더명 영문 변경 |
| `google-generativeai` deprecated | 패키지 지원 종료 | `google-genai` 패키지로 전환, API 호출 코드 변경 |
| 벡터 DB 모드별 분리 vs 단일화 | 임베딩 모델별 벡터 공간 불일치 | 임베딩을 bge-m3로 통일, LLM만 모드 전환하여 단일 컬렉션 공유 |
