# 은행회계RAG 개선/튜닝 계획

> 본 문서는 은행회계 RAG 시스템의 **답변 정답율을 높이기 위한 개선·튜닝 작업** 전반(평가 인프라 구축 → 검색 튜닝 → 프롬프트 엔지니어링 → 필요시 모델 파인튜닝)을 추적한다.

## 1. 배경 및 목적

- 현재 은행회계 RAG(`banking_accounting_rag`) 시스템은 로컬 Ollama / Gemini 2.5 Flash / OpenAI GPT-4o 세 가지 LLM 모드를 지원하며, bge-m3 임베딩 기반 ChromaDB를 사용한다.
- 서비스 품질을 높이려면 **답변 정답율을 정량적으로 측정**하고, 측정 결과를 기반으로 **검색/생성 파이프라인을 튜닝**해야 한다.
- "얼마나 정확한가?"에 대한 수치 근거 없이 튜닝하면 체감에만 의존하게 되어, 변경이 개선인지 퇴보인지 판단이 불가능하다. 따라서 **평가 인프라를 최우선으로 구축**한다.

## 2. 전체 로드맵 (4단계)

| 단계 | 이름 | 목표 | 상태 |
|------|------|------|------|
| 1 | **평가 기반 구축** | 골든 Q&A 데이터셋 + 자동 채점 파이프라인 확보 | 🟡 MVP 구현 완료 / 실행 대기 |
| 2 | **검색(Retrieval) 품질 개선** | 청크 전략·하이브리드 검색·리랭킹으로 Hit@K 향상 | ⬜ 미착수 |
| 3 | **프롬프트 엔지니어링** | 시스템 프롬프트·Few-shot 예시로 답변 품질 향상 | ⬜ 미착수 |
| 4 | **모델 파인튜닝** | 임베딩/LLM 파인튜닝 (비용 대비 효과 확인 후) | ⬜ 미착수 |

**진행 원칙:** 각 단계는 이전 단계의 평가 지표로 A/B 비교해야 의미가 있다. 1단계 없이 2·3단계로 넘어가지 않는다.

## 3. 1단계: 평가 기반 구축

### 3.1 목표

- 사용자 질문과 유사한 분포의 골든 Q&A 세트를 확보한다.
- 신규 변경(청크 크기 조정, 프롬프트 수정 등)이 있을 때마다 **한 번의 명령으로 정답율을 측정**할 수 있는 자동화 파이프라인을 갖춘다.

### 3.2 구현 결과물 — 평가 에이전트 (`scripts/eval_agent/`)

4단계 스크립트로 구성된 파이프라인. 각 단계는 독립 실행 가능하며 jsonl 파일로 중간 산출물을 저장한다.

| 스크립트                   | 역할                                            | 입력                  | 출력                                  |
| ---------------------- | --------------------------------------------- | ------------------- | ----------------------------------- |
| `step1_generate_qa.py` | `docs/md` 페이지 샘플링 후 Gemini 2.5로 Q&A 쌍 자동 생성   | 마크다운 원문             | `data/qa_dataset.jsonl`             |
| `step2_run_rag.py`     | 생성된 질문을 `rag_chain.ask()`에 전달, 답변 + 검색 청크 수집  | `qa_dataset.jsonl`  | `data/rag_results.jsonl`            |
| `step3_judge.py`       | LLM-as-a-judge (Gemini 2.5) 로 0~3점 채점 + 근거 저장 | `rag_results.jsonl` | `data/judgments.jsonl`              |
| `step4_report.py`      | 집계 및 마크다운/CSV 리포트 생성                          | `judgments.jsonl`   | `data/report.md`, `data/report.csv` |
| `run_all.py`           | 전체 파이프라인 순차 실행                                | —                   | —                                   |

### 3.3 Q&A 생성 전략

- 3가지 유형을 순환 배정:
  - **factual** — 페이지에 직접 서술된 정의/수치/규정
  - **reasoning** — 개념 비교·원리 설명 (종합 추론 요구)
  - **judgment** — 실무 상황 가정 시 올바른 회계처리/판단
- 각 Q&A에 출처 페이지(`source_volume`, `source_page`) 메타데이터 기록 → Hit@K 계산에 사용

### 3.4 평가 지표

| 지표 | 정의 |
|------|------|
| **평균 점수** | judge 점수 평균 (0~3) |
| **정답율** | 2점(대체로 정답) 이상 비율 (%) |
| **Hit@K** | 검색 상위 K개 청크에 골드 페이지가 포함된 비율 (%) |

정답율과 Hit@K를 분리해 보는 이유: 답변이 틀렸을 때 **검색이 실패했는지(retrieval 문제) / 검색은 맞았는데 LLM이 답을 못 만들었는지(generation 문제)** 원인 분리가 가능해야 튜닝 방향이 명확해진다.

### 3.5 MVP 설정 (`scripts/eval_agent/eval_config.py`)

| 항목 | 값 |
|------|-----|
| 대상 문서 | 은행회계해설(상) p.41~140 (제3장 재무상태표 계정과목 초반) |
| 샘플 페이지 | 10개 (seed=42 고정) |
| 페이지당 Q&A | 2개 × 3유형 순환 → **총 약 20문제** |
| 생성/채점 모델 | `gemini-2.5-flash` |
| 평가 대상 RAG 모드 | `gemini` |

**작게 시작하는 이유:** 생성된 Q&A 품질이 나쁘면 그 위의 모든 분석이 의미 없어진다. 20문제 정도로 프롬프트/루브릭 품질을 눈으로 검증한 뒤 확장한다.

### 3.6 알려진 한계 및 대응

| 리스크 | 설명 | 완화 방법 |
|--------|------|----------|
| 자동 생성 Q&A가 얄팍함 | "책에 뭐라고 적혀 있는가" 수준의 표면 질문만 양산될 수 있음 | (a) 유형별 프롬프트에 구체적 예시 명시 (b) 향후 사용자가 수동으로 만든 씨드 질문과 병합 |
| LLM-as-a-judge 편향 | 장황하고 자신감 있는 답변에 후한 점수 경향 | 20문제 중 5~6개를 사람이 재채점해 judge 신뢰도 검증. 루브릭 문구를 반복 조정 |
| Gemini API rate limit | 짧은 시간에 많은 호출 시 429 | 각 호출 간 0.3~0.5초 sleep, 무료 티어 일일 한도(1,500회) 내 운용 |

## 4. 2단계: 검색(Retrieval) 품질 개선 (예정)

1단계로 베이스라인을 확정한 뒤 착수. 후보 작업:

- 청크 전략 튜닝: `CHUNK_SIZE` 1000 → 500/1500 비교
- **하이브리드 검색**: 벡터 검색 + BM25 키워드 검색 병합 (회계 용어/숫자/계정과목은 키워드 매칭이 중요)
- **리랭킹**: bge-reranker-v2-m3 Cross-Encoder로 TOP 20 → TOP 5 재정렬
- 쿼리 재작성: 사용자 질문을 검색 친화적으로 사전 변환

**평가 방법:** 각 변경마다 `eval_agent/run_all.py` 재실행 → `report.md`의 Hit@K·정답율 비교.

---

## 5. 3단계: 프롬프트 엔지니어링 (예정)

- 시스템 프롬프트 가드레일 강화 (환각 방지, 인용 강제)
- Few-shot 예시 삽입 (좋은 답변 2~3개)
- 답변 형식 정형화 (근거 인용 블록 필수 등)

---

## 6. 4단계: 파인튜닝 (보류)

- 임베딩 모델 LoRA 파인튜닝 (bge-m3, 도메인 Q&A 기반)
- LLM LoRA 파인튜닝 (gemma3:12b)
- **착수 조건:** 2·3단계까지 진행해도 정답율이 목표치(예: 80%)에 못 미칠 때. 투자 대비 효과가 떨어지는 경우가 많아 마지막 카드로 보류.
